## 2. API通信验证 + 平台规则采集

**目标**：对每个平台，确认三件事：
1. 有没有API可以用（技术接入）
2. 平台规则是什么（商业合规）
3. 文档/手册在哪里（知识入库）

### 2.1 验证流程（双线并行）

每条线独立跑完后合并结果：

```yaml
# 线A: API技术验证
api_checklist:
  1. 找开发者文档
  2. 确认：API存在否 / 类型 / 认证 / 费用 / 限速 / 数据范围
  3. 实际测试（curl测试端点）
  4. 记录文档URL

# 线B: 平台规则采集
rules_checklist:
  1. 找卖家/创作者/商户帮助中心
  2. 采集以下文件（PDF/HTML）：
     - 使用条款 (Terms of Service)
     - 卖家/创作者协议
     - 费用结构 / 佣金比例
     - 禁售品 / 受限商品清单
     - 内容审核规则 / 社区指南
     - 纠纷处理流程 / 退款政策
     - 入驻要求 / KYC标准
     - 合规认证要求（品类特定）
     - API使用条款（独立于技术文档）
  3. 判断是否有规律更新的Changelog/Policy Change页面
  4. 记录所有文档URL
```

### 2.2 验证结果分类（扩展版）

| 标记 | 含义 |
|------|------|
| ✅ 官方API可用 | 有文档，能接入，知道成本和限速 |
| ⚠️ API存在但受限 | 要申请/付费/审核/特定资质 |
| ❓ 未找到公开API | 没有官方API文档 |
| ❌ 明确无API | 官方声明不提供 |
| 📡 RSS可用 | 没有API但有RSS |
| 🔮 需人工确认 | 非英文/无信息/需本地账号 |
| 📖 文档已采集 | 规则文件/帮助手册已入库 |
| ❓ 规则未找到 | 找不到平台的卖家/商户文档 |

### 2.3 输出格式

每个平台的验证结果汇总为一行：

```csv
region,platform,category,url,has_api,api_type,pricing,rate_limit,scopes,data_scope,docs_url,status
Russia,Ozon,ecommerce,ozon.ru,Yes,REST+JSON,Free+Paid,100/min,products/prices/stock,商品/价格/库存,docs.ozon.ru/api,verified
```

## Phase 3：接入决策

**目标**：基于Phase 2的结果，决定每个平台的最终接入方式。

### 3.1 决策树

```
平台是否有官方API？
├─ 是
│  ├─ 免费且不封号 → 直接接入（第一层）
│  ├─ 付费但数据有价值 → 接入（第一层，算成本）
│  └─ 要审核且过不了 → 降级
│
├─ 否，但有RSS
│  └─ 接RSS（第二层·免费开源）
│
└─ 否，完全无API
   ├─ 页面结构稳定 → 走爬虫（第二层·低风险）
   ├─ 页面结构不稳定 → 走付费聚合（第三层）
   └─ 爬虫风险高（封IP） → 直接付费聚合
```

### 3.2 三层采集优先级（确认版）

| 层级 | 策略 | 适用场景 | 风险 |
|------|------|---------|------|
| **L1：官方API** | 官方开发者API | 有文档、免费或合理付费 | 低（有合同约束） |
| **L2：免费开源** | RSS / 静态爬虫 / 公开数据集 | 无API但有公开信息 | 中（注意封IP） |
| **L3：付费聚合** | RedFox / 第三方数据API | 国内平台 / 无公开API | 低（付费有服务保障） |
| **L4：人工兜底** | 标记+手动检查周期 | 实在接不了的 | 高（不自动化） |

> L4 应当尽可能少。每个 L4 平台都意味着情报盲区。

### 3.3 输出

```yaml
# 每个平台的最终接入方案
platform: NSF International
final_decision: website_monitoring  # api / rss / scraper / paid_aggregator / manual
tier: L2                            # L1/L2/L3/L4
collector_type: rss_checker         # 采集器模块名
refresh_frequency: daily            # daily / weekly / monthly / event_driven
priority_score: 45                  # 影响×频率×时效
cost_per_month: 0                   # 每月费用
status: active                      # active / planned / paused / blocked
notes: "无官方API，监控NSF标准页面+邮件列表"
```

**目标**：按优先级分批接入。

## Phase 6：持续运营

| 频率 | 事项 |
|------|------|
| 每日 | 检查日报覆盖率和采集成功率 |
| 每周 | 检查新平台API变化（之前不可用的可能变可用了） |
| 每月 | 调整优先级评分（市场变化了，权重要调） |
| 每季 | 新增平台扫描（新平台出现/老的关了） |
| 事件 | 采集器挂了→自动降级→人工修复 |

---

## SOP执行原则

1. **宁可不接，不要瞎接** — 数据质量 > 数据数量
2. **每步确认再走下一步** — 先验证API再入库，先搭架子再铺量
3. **不可污染** — 原始数据=原始数据，分析=分析，两码事
4. **降级策略优先于主策略** — 任何接入方案都必须有B计划
5. **成本透明** — 每个平台的接入费用（API订阅/代理/IP/时间）都要算清楚
